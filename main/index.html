# LangSmith Alerts：提前捕获生产故障

可靠的应用是优质用户体验的基础，而提前发现故障是关键。LangSmith 推出了 Alerts 功能，帮助开发者实时监控 LLM 应用和代理的表现。支持基于 错误率、运行延迟 和 反馈评分 等关键指标设置警报。

为什么主动监控很重要？

LLM 应用面临两大挑战：

  1. 外部服务依赖：模型提供商、API、数据库等依赖的故障或延迟会严重影响体验。
  2. 输出质量与正确性：提示、模型或输入的微小变化可能导致结果不稳定，反馈评分警报可提供早期预警。

LangSmith Alerts 概览

支持以下指标的警报：

  • 错误数量与比例
  • 平均延迟
  • 平均反馈评分

可通过筛选条件（如模型、工具调用类型）聚焦特定运行子集，并设置时间窗口（5 或 15 分钟）和阈值。集成方式包括 PagerDuty 或自定义 Webhook（如 Slack 通知）。

# 关于智能体框架的思考总结

本文探讨了智能体系统（agentic systems）的设计与实现，分析了智能体框架的关键挑战和特性，并提出了对现有框架的评价标准。

核心观点

  1. 可靠智能体系统的难点
    构建可靠的智能体系统的关键在于确保语言模型（LLM）在每一步都获得适当的上下文。这包括控制输入内容以及生成相关上下文的步骤。

  2. 智能体与工作流的关系
    智能体系统包含工作流（workflows）和智能体（agents），以及两者的混合形式。大多数生产环境中的系统是二者的结合。

  3. 智能体框架的本质
    大多数智能体框架并非声明式或命令式编排框架，而是提供了一组智能体抽象（agent abstractions）。这些抽象虽然易于上手，但可能掩盖了上下文传递的复杂性。

  4. LangGraph 的定位
    LangGraph 是一个编排框架，支持声明式和命令式 API，并在其基础上构建了智能体抽象。它提供了灵活的低层功能，同时支持高级用例。

智能体框架的核心维度

  1. 工作流 vs 智能体
  • 工作流：更可预测，适合定义明确的任务。
  • 智能体：灵活性更高，适合需要动态决策的场景。
  • 实际应用中，大多数系统是两者的结合。
  1. 声明式 vs 命令式
  • LangGraph 采用声明式图结构表示节点和边，但底层逻辑仍为正常的代码实现。
  • 其他框架（如 Agents SDK）仅提供智能体抽象，而非真正的编排框架。

  1. 多智能体通信
    多智能体系统的关键在于智能体之间的通信机制。通过工作流连接智能体可以提高可靠性。

  2. 框架的价值

  • 提供短期记忆、长期记忆、人类介入（human-in-the-loop）、流式处理等功能。
  • 支持调试、可观测性和容错机制。

常见问题解答

  1. 是否需要智能体框架？
    如果应用不需要复杂的生产功能（如持久化、容错等），可以不使用框架。但对于复杂场景，框架能显著提升开发效率和系统可靠性。

  2. 模型进步后，是否会取代工作流？
    即使模型性能提升，控制上下文传递仍然重要。对于特定任务，简单的工具调用循环可能足够；但对于大多数场景，工作流和智能体的结合仍是最佳实践。

  3. OpenAI 的观点有何问题？
    OpenAI 的指南混淆了“声明式 vs 命令式”、“工作流 vs 智能体”等概念，低估了可靠编排框架的重要性。

结论

  • 可靠智能体系统的核心在于确保 LLM 获得正确的上下文。
  • 不同框架在灵活性、易用性和功能性上各有优劣。
  • LangGraph 提供了一个兼顾低门槛和高扩展性的解决方案，适合构建生产级智能体系统。
