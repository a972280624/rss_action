  • How to think about agent frameworks

    # 如何思考智能体框架 本文探讨了构建可靠智能体系统的核心挑战及智能体框架的设计理念，以下是总结： --- ## 核心观点 1. **构建可靠智能体系统的难点** 确保大语言模型（LLM）在每个步骤中都能获得正确的上下文是关键。这包括控制输入到模型的内容，以及执行适当的步骤生成相关内容。 2. **智能体系统的组成** 智能体系统既包含工作流（workflows），也包含智能体（agents），甚至两者的混合形式。 3. **现有智能体框架的局限性** 大多数智能体框架只是提供了一些智能体抽象，并非真正的声明式或命令式编排框架。这种抽象虽然易于上手，但可能会掩盖上下文管理的复杂性。 4. **LangGraph 的定位** LangGraph 是一个兼具声明式和命令式 API 的编排框架，支持工作流、智能体及其混合形式。它旨在为开发者提供对 LLM 上下文的精确控制，同时解决生产环境中的持久化、容错等问题。 --- ## 背景信息 ### 什么是智能体？ OpenAI 和 Anthropic 对智能体的定义各有侧重： - **OpenAI**：强调智能体是能够独立完成任务的系统。 - **Anthropic**：更技术性地将智能体定义为动态使用工具并根据反馈调整行为的系统。 智能体与工作流的区别在于，前者具有更高的灵活性和自主性，而后者更注重预定义流程的可靠性。 ### 构建智能体的难点 构建可靠的智能体系统之所以困难，主要是因为： - 提供给 LLM 的上下文不完整或不准确。 - 工具描述模糊、用户输入不明确等问题都会导致性能下降。 --- ## 智能体框架的类型 1. **工作流 vs 智能体** - 工作流：更适合确定性和可预测性要求高的场景。 - 智能体：适合需要灵活性和大规模决策的场景。 2. **声明式 vs 命令式** LangGraph 提供了声明式和命令式两种接口，开发者可以根据需求选择。 3. **多智能体系统** 在复杂任务中，多个智能体通过通信协作可以提高性能。如何设计通信机制是关键。 --- ## 常见问题解答 1. **框架的价值是什么？** 框架不仅提供智能体抽象，还支持短时记忆、长时记忆、人类介入、调试等功能，帮助开发者更高效地构建可靠系统。 2. **随着模型进步，所有系统都会变成智能体吗？** 不一定。尽管模型性能会提升，但对于特定任务，工作流可能仍然是更简单、更高效的解决方案。 3. **OpenAI 的观点有何问题？** OpenAI 的指南混淆了“声明式 vs 命令式”和“智能体 vs 工作流”的概念，忽略了编排层的重要性。 --- ## 总结 智能体系统的设计需要平衡灵活性与可靠性，LangGraph 提供了一个低门槛、高天花板的框架，适用于各种智能体和工作流的组合场景。
